권장 연동 구조

외부 업체는 가능하면 Next.js API 기준으로 연동하고, Stable 백엔드와 Payaction, thirdweb 관련 처리 및 비밀키 관리는 서버 내부에 두는 방식을 권장합니다.

외부 업체 Next.js API Stable / Payaction / thirdweb

핵심 운영 포인트

  • 현재 결제 페이지에서는 스마트 어카운트 연결 없이도 구매 진행이 가능합니다.
  • 핵심 주문 생성 API는 POST /api/order/setBuyOrder 입니다.
  • 결제요청은 서버가 https://api.payaction.app/order 를 호출합니다.
  • 결제확정은 thirdweb SDK 또는 thirdweb engine 으로 처리됩니다.
권장 전달 문구

현재 결제 연동은 웹에서 Next.js API 를 호출하고, 서버가 Stable 백엔드와 Payaction, thirdweb 관련 처리를 순차 수행하는 구조입니다. 외부 연동은 우선 /api/order/setBuyOrder 중심으로 맞추고, 결제요청과 결제확정은 서버 내부 처리로 두는 방식을 권장합니다.

전체 흐름

현재 payment 페이지 기준 구매 여정입니다.
1

클라이언트 정보 조회

결제 체인과 기본 환율 정보를 확인합니다.

2

가맹점 정보 조회

가맹점 운영 상태와 결제 한도를 읽어옵니다.

3

구매자 생성 또는 조회

입금자 정보 기반으로 구매자 지갑 주소를 확보합니다.

4

기존 주문 확인

같은 사용자의 진행 중 구매 주문이 있는지 확인합니다.

5

판매 주문 매칭

같은 KRW 금액의 오픈 판매 주문이 있으면 즉시 수락합니다.

6

구매 주문 생성

매칭 주문이 없으면 신규 구매 주문을 생성합니다.

주문 생성 후 이동

주문 생성이 완료되면 /pay-usdt-reverse/[orderId] 페이지로 이동하여 후속 확인을 진행합니다.

후속 서버 처리

필요 시 서버가 결제요청, 결제확정, 감사로그 저장까지 이어서 처리합니다.

실무상 핵심 포인트

외부 업체가 직접 맞추면 되는 핵심은 구매자 생성과 주문 생성 구간입니다. Payaction 연동 키, thirdweb 비밀키, 엔진 토큰은 서버 내부에만 보관하는 구조가 안전합니다.

Next.js API 기준 연동 권장

  • 외부 업체는 내부 Stable 백엔드 주소를 몰라도 됩니다.
  • Payaction, thirdweb 관련 키를 외부에 노출하지 않습니다.
  • 운영 예외처리와 감사로그를 서버에 유지할 수 있습니다.

직접 백엔드 연동 시 주의

  • 주문 생성 외의 후속 처리 범위를 별도로 정의해야 합니다.
  • 인증 체계와 허용 IP 정책이 추가로 필요할 수 있습니다.
  • 응답 포맷 일부가 API 별로 완전히 동일하지 않습니다.

서버 내부 처리 1: 결제요청

POST /api/order/requestPayment 는 서버가 https://api.payaction.app/order 를 대신 호출하는 구조입니다. x-api-key 와 x-mall-id 는 서버 내부에서만 관리해야 합니다.

서버 내부 처리 2: 결제확정

POST /api/order/confirmPayment 는 thirdweb SDK 또는 thirdweb engine 을 통해 에스크로 전송을 수행합니다. 외부 업체에 공개할 공개 API로 보기보다는 서버 내부 오케스트레이션 API 로 보는 것이 적절합니다.

보안 체크리스트

Payaction API Key, Mall ID, THIRDWEB_SECRET_KEY, THIRDWEB_ENGINE_ACCESS_TOKEN, DB 접속 정보는 반드시 서버에서만 보관해야 합니다. 외부 업체에는 가능하면 Next.js API 만 개방하는 방식을 권장합니다.
